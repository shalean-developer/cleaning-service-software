# Recurring booking engine — production soak plan

Stage 4 operational hardening before monthly invoicing, auto-charge, or advanced billing.

**Scope:** recurring series, generated child visits, per-visit payment, cleaner visibility. Does **not** change one-off booking flows.

## Prerequisites

- Stage 2 engine deployed (`booking_series`, `bookings.series_id`)
- `CRON_SECRET` configured on app and scheduler
- Cron route documented: `docs/operations/generate-recurring-occurrences-cron.md`
- Ops commands available:
  - `npm run ops:audit:recurring-bookings`
  - `npm run ops:soak:recurring-bookings`

## Launch checklist

- [ ] `npm run ops:audit:recurring-bookings` → Issues: 0
- [ ] `/admin/recurring/health` → overall PASS (or WARN only with documented acceptance)
- [ ] Cron manually triggered once; `recurring_generation_runs` row present
- [ ] Weekly / biweekly / monthly test series exercised (below)
- [ ] Customer “Pay next visit” completes checkout for a generated child
- [ ] Cleaner does **not** see unpaid generated child
- [ ] Cleaner sees child only after paid + dispatch-ready status
- [ ] Pause / resume / cancel behave as expected
- [ ] No monthly invoicing or auto-charge code paths enabled

---

## 1. Weekly series

1. Create or use an active weekly series (first visit paid, series materialized).
2. Trigger cron (or wait for schedule).
3. **Expected DB:**
   - `booking_series.status = active`, `frequency = weekly`
   - Child rows: `series_id` set, `status = pending_payment`, `metadata.recurring.generated = true`
   - Unique `(series_id, scheduled_start)` — no duplicates
4. Customer pays next visit via recurring detail → Paystack retry/checkout.
5. **Expected after payment:**
   - Child `status` → `confirmed` (then assignment pipeline)
   - Paid row in `payments` for child `booking_id`
6. **Cleaner visibility:** no row in cleaner feed until dispatch-ready paid status.

## 2. Biweekly series

Same as weekly with `frequency = biweekly` and +14 day JHB slots.

Verify `next_occurrence_at` advances correctly after generation.

## 3. Monthly series

Use anchor on day 29–31 if available; confirm month-end clamp (audit may WARN `MONTHLY_EOM_EDGE` — review only).

## Expected child statuses

| Stage | Booking status | Payment |
|-------|----------------|---------|
| Generated by cron | `pending_payment` | uninitialized/pending |
| Customer checkout started | `pending_payment` | pending |
| Paid | `confirmed` → assignment | `paid` |
| Abandoned checkout | `pending_payment` or `payment_failed` | per expiry cron |
| Paused series | no **new** children | existing unpaid may remain |

## Payment behavior (per-visit)

- Each generated child is a separate booking + payment.
- “Pay next visit” uses `startPaymentRetryCheckout` (same as payment retry elsewhere).
- Payment finalize must target the **child** `booking_id`, not the anchor only.
- Failed payment must **not** start assignment dispatch.
- Idempotent finalize must not duplicate dispatch offers.

## Cleaner visibility

- Unpaid generated children: **hidden** from cleaner assignment views.
- Audit flag `UNPAID_CHILD_CLEANER_VISIBLE` must be zero.
- After pay + dispatch: cleaner sees offer/assignment per existing rules (unchanged).

## Pause / resume / cancel

| Action | Expected |
|--------|----------|
| Pause | `booking_series.status = paused`; cron does not create new children |
| Resume | `active`; cron resumes generation |
| Cancel | `cancelled`; no new children; unpaid children may remain (WARN) |

## Skip / reschedule (admin)

- Skip next: advances series pointer; audit `RECURRING_SERIES_SKIP_NEXT`
- Reschedule next: updates `next_occurrence_at`; audit `RECURRING_SERIES_RESCHEDULE_NEXT`

## Audit commands

```bash
npm run ops:audit:recurring-bookings
npm run ops:soak:recurring-bookings
```

Review `/admin/recurring/health` for PASS/WARN/FAIL summary cards.

## Rollback plan

1. **Pause cron** in Vercel or disable pg_cron job — stops new children (existing rows remain).
2. **Pause series** via admin UI for affected customers.
3. Do **not** bulk-delete `booking_series` or children in production without ops playbook.
4. Payment/assignment issues: use existing admin recovery cron and support tools (unchanged).
5. Re-enable cron after fix; generation is idempotent (duplicates skipped).

## Remaining limitations (Stage 4)

- No monthly consolidated invoicing
- No auto-charge / saved card recurring billing
- Batch generation limited to 100 active series per run (`listActiveSeriesForGeneration`)
- Health dashboard queries capped (200 series / 500 bookings in horizon)
